## Item 1 — Pre-commit framework: shareable hook + local gate

### Current
`.git/hooks/pre-commit` (unversioned, generated by `gga install`):
```bash
gga run || exit 1   # ONLY GGA — no lint/vet/build
```
Fresh clones get no hook. GGA has its own `gga install` hook manager (writes to
`.git/hooks/`). No `.pre-commit-config.yaml`, no lefthook config, no `core.hooksPath`.

### Approaches
| Approach | Pros | Cons | Effort |
|---|---|---|---|
| **A. pre-commit (Python)** | `.pre-commit-config.yaml` versioned; official golangci-lint hook; large ecosystem | Adds **Python runtime dep** — violates AGENTS.md "prefer Go stdlib"; GGA is custom (needs local-hook wrapper); heavier for a Go project | Medium |
| **B. lefthook (Go binary)** | Single Go binary; fast/parallel; `.lefthook.yml` versioned; glob filtering; calls `gga run` as a command | New binary dep (brew/go install lefthook) for a small team; overlaps GGA's own hook manager | Medium |
| **C. Versioned shell script + `core.hooksPath`** | **Zero new deps**; versioned `.githooks/pre-commit`; runs full local gate (GGA + `golangci-lint run` + `go vet ./...` + `go build ./...`); `task setup` runs `git config core.hooksPath .githooks`; matches "prefer stdlib/minimal deps" ethos | No parallelism/glob filtering; each dev runs a one-time `task setup`; must not also run `gga install` (conflicts with hooksPath) | Low |

### Recommendation
**Approach C** — versioned `.githooks/pre-commit` + `git config core.hooksPath .githooks`
via a new `task setup`. It is the only option with zero new dependencies (AGENTS.md
"MUST justify any new dependency", "MUST NOT add dependencies for trivial
functionality"). The script invokes `gga run` directly (bypassing `gga install`'s
`.git/hooks` writer) plus the deterministic local gate, fixing gap #7. Document the
one-time `task setup` in CONTRIBUTING.md "Setup". lefthook (B) is the polished
alternative if the team later wants parallelism; pre-commit (A) is the worst fit
(Python dep).

### Risks
- Devs forgetting `task setup` on fresh clone (no hook) — Medium. Mitigate: document
  in CONTRIBUTING + add a `task setup` note to PR template.
- `core.hooksPath` + `gga install` conflict — Low. Mitigate: script calls `gga run`
  directly; do not run `gga install` when hooksPath is set.

## Item 2 — Per-package 80% coverage gate for `internal/`

### Current
`task cover` measures TOTAL (75% threshold). AGENTS.md mandates per-package 80% for
`internal/`. Not enforced in CI. See measured table above: all `internal/` currently
≥80% (gap latent), `cmd/` 49% by-design.

### Approaches
| Approach | Pros | Cons | Effort |
|---|---|---|---|
| **A. Custom shell/awk parsing `go test -cover ./...`** | **No new dep**; gates `internal/**` at 80%, excludes `cmd/`/main/testutil; ~15 lines in Taskfile | Hand-rolled parsing; must handle `ok`/`FAIL`/`[no statements]` lines | Low |
| **B. `go-test-coverage` / `gotestum` tool** | Battle-tested threshold logic | **New dependency** — must justify vs stdlib per AGENTS.md | Medium |
| **C. Codecov component/flag gates** | Server-side gating per-package; trend graphs | External service dependency; requires Codecov config + token; current codecov-action has `fail_ci_if_error: false` | Medium-High |

### Recommendation
**Approach A** — add a Taskfile `cover:pkg` command that runs
`go test -cover ./...`, greps `coverage:` lines, and fails if any package matching
`internal/` reports `< 80.0%`. Output format is stable:
`ok  github.com/.../internal/backup  (cached)  coverage: 83.1% of statements`.
Exclude `cmd`, root, `config/testutil` (test helper, 0% by design), and `[no statements]`.
Wire it into the CI `coverage` job alongside the existing total gate (keep 75% total as
a floor, add 80% per-internal as the real gate). No new dependency — satisfies AGENTS.md.
Optionally lower the total threshold note since per-package becomes the meaningful gate.

### Risks
- `tui/screens` at exactly 80.0% will flake on any regression — Medium. This is the
  POINT (it enforces AGENTS.md); first task in apply may need a tiny coverage bump.
- Shell parsing edge cases (cached vs uncached, `[no statements]`) — Low. Mitigate:
  test the awk locally in apply.

## Item 3 — golangci-lint-action vs `go install @latest`

### Current (ci.yml lint job)
```yaml
- run: go install github.com/golangci/golangci-lint/v2/cmd/golangci-lint@latest
- run: golangci-lint run
```
`@latest` is **unpinned** — a new golangci-lint release changing defaults can break CI
overnight with no code change. Source build is also slow (compiles golangci-lint).

### Approaches
| Approach | Pros | Cons | Effort |
|---|---|---|---|
| **A. `golangci/golangci-lint-action@v8`** | Pins `version: v2.12.2`; `install-mode: binary` (default) downloads prebuilt — **no compile**; built-in cache (`~/.cache/golangci-lint`); `verify: true` validates `.golangci.yml` for free; v8 works with golangci-lint v2.1+ on Node 20 | Action version must be maintained; Node 20 runner deprecation looming (v9 needs Node 24) | Low |
| **B. `golangci/golangci-lint-action@v9`** | Latest; Node 24 (forward-looking); same pinning/cache/binary benefits | Requires Node 24 runner; newer = slightly less battle-tested | Low |
| **C. Keep `go install` but pin `@v2.12.2`** | Minimal diff; no action dependency | Still compiles from source (slow); no built-in cache; no config verify | Low |

### Recommendation
**Approach A** (`@v8`) with `version: v2.12.2` (exact patch, matching the locally
installed v2.12.2), `install-mode: binary` (default). Replaces the source build with a
pinned prebuilt binary + free cache + config validation. **Correction to the brief:**
the orchestrator referenced `@v6` — v6 predates v2-only support; the correct modern
pinnings are v8 (Node 20, safe) or v9 (Node 24, latest). Pin `@v8` now; note v9 as the
migration target once Node 24 is standard. Keep `actions/setup-go@v5` before it (v8
requires a setup-go step). `verify: true` (default) will catch `.golangci.yml` schema
errors in CI — a bonus once the new refactoring linters are added (Item 8).

### Risks
- Action pinning drift — Low. Mitigate: pin exact `v2.12.2`; bump deliberately.
- Node 20 deprecation eventually forces v9 — Low. Document in CHANGELOG when moved.

## Item 4 — govulncheck/gosec: stop swallowing failures

### Current (Taskfile `security`)
```yaml
- govulncheck ./... || true
- gosec -quiet ./... || true
```
Both swallow. For a tool that handles GitHub OAuth tokens, a real CVE in a reachable
dependency passes green.

### Approaches
| Approach | Pros | Cons | Effort |
|---|---|---|---|
| **A. govulncheck blocks, gosec advisory** | govulncheck does source-reachability analysis → **very low FP rate** (high-signal); gosec has known FPs (G301/G306/G304 already excluded) → keep advisory | A newly-disclosed transitive CVE can fail CI mid-cycle (desirable but noisy) | Low |
| **B. Both block** | Strongest posture | gosec FPs will block legit PRs; noisy for small team | Low |
| **C. Keep both advisory** | Zero CI friction | Real vulns don't block — unacceptable for a token-handling tool | None |

### Recommendation
**Approach A** — `govulncheck ./...` **blocks** (remove `|| true`); `gosec -quiet ./...`
stays **advisory** (`|| true` kept, prints warnings). Rationale: govulncheck traces
actual reachable call paths (low FP), so a failure means a real exploitable path in a
dependency bak-cli actually uses — must block for a token-handling tool. gosec is
static/heuristic with documented FPs already excluded; keep it as a warning signal.
Also: pin `govulncheck@v1.x` (currently `@latest`) and `gosec@v2.x` for reproducibility.
Split into `task security` (advisory gosec) and make the CI `security` job fail on
govulncheck non-zero.

### Risks
- New CVE in a dep breaks CI before bak-cli can bump — Medium. This is the correct
  security posture; mitigate by bumping the dep in the same PR.
- govulncheck reachability false-negatives on rarely-tested paths — Low. Acceptable.

## Item 5 — GGA CI install: drop brew-on-Linux

### Current (gga.yml)
```yaml
- run: |
    brew update
    brew install gga
```
On ubuntu-latest this bootstraps linuxbrew + tap update — slow and fragile.

### Findings
GGA source: `github.com/Gentleman-Programming/gentleman-guardian-angel` (MIT). The brew
formula builds **from a source tarball**
(`.../archive/refs/tags/v2.8.1.tar.gz`) — i.e. GGA is a Go project whose source compiles.
GGA exposes `gga --ci` (review HEAD~1..HEAD), `--pr-mode`, `--diff-only`. Local install
is linuxbrew v2.8.1.

### Approaches
| Approach | Pros | Cons | Effort |
|---|---|---|---|
| **A. `go install <module>/cmd/gga@v2.8.1`** | Uses Go already in the job; **no linuxbrew bootstrap**; pinned; fast compile | Must confirm exact module/cmd path against the GGA repo layout (brew proves source compiles, not the `go install` import path) | Low |
| **B. GitHub release binary (curl)** | Fastest (no compile) | **Unconfirmed** whether GGA publishes prebuilt release binaries (brew builds from source → may not exist) | Low |
| **C. Keep brew** | Already works | Slow (~30-60s `brew update`), fragile on Linux, large bootstrapped env | None |

### Recommendation
**Approach A** — `go install` the GGA binary pinned to `v2.8.1` (matching local). The CI
job already sets up Go, so this removes the entire linuxbrew dependency. **Apply-phase
prerequisite:** confirm the exact `go install` import path by inspecting the GGA repo's
`cmd/` layout (likely `github.com/Gentleman-Programming/gentleman-guardian-angel/cmd/gga`
or the repo root). If `go install` fails (non-standard module layout), fall back to B
(if releases exist) or a `go build` from the tagged tarball. Approach C (brew) is the
fallback-of-last-resort, not the default.

### Risks
- GGA module path differs from brew source layout → `go install` fails — Medium.
  Mitigate: verify path in apply before committing the workflow change.
- GGA may not tag the cmd path consistently across versions — Low. Pin exact `v2.8.1`.

## Item 6 — GGA gate vs advisory

### Current
`gga.yml` job has `continue-on-error: true` → GGA never blocks merge. This is
**REQ-CI-004** (ci-consistency spec, `should` priority): "MUST include a non-blocking
GGA review job ... MUST NOT block merge (warn-only)". AGENTS.md rule #41 makes the
**local pre-commit** the enforcement point with the `NO-VERIFY:` bypass clause
(`gga-bypass` capability).

### Findings (historical evidence)
The archive shows GGA has **repeatedly failed on technical grounds**:
- `2026-06-09-v2-migration`: GGA crashes on Windows with "Argument list too long" (ARG_MAX).
- `2026-06-16-coverage-improvement`: GGA hook **corrupted the git index**, required
  `git rm -r --cached .` recovery; final commit used `--no-verify`.
- `2026-06-18-gga-pr-merge`: PR #26 `--no-verify` due to ARG_MAX on 11 files.
- `2026-06-18-critical-fixes`: `NO-VERIFY:` used for pre-existing table-driven test violations.

GGA is an **AI reviewer** (provider `opencode:opencode-go/qwen3.7-plus`) with false
positives and infrastructure flakiness. `.gga` has `STRICT_MODE="true"` but no
severity field — GGA does not expose a severity taxonomy to gate on.

### Approaches
| Approach | Pros | Cons | Effort |
|---|---|---|---|
| **A. Keep advisory (status quo, REQ-CI-004)** | Matches the deliberate prior decision; doesn't block on GGA flakiness; local pre-commit + `NO-VERIFY:` remains the honest enforcement point | GGA CI findings can be ignored | None |
| **B. Hard gate (block on GGA fail)** | AGENTS.md violations can't merge | **Contradicts REQ-CI-004** (needs MODIFIED delta + rationale); blocks PRs on GGA's demonstrated flakiness (ARG_MAX, index corruption, provider outage) | Medium |
| **C. Hybrid: GGA advisory + deterministic teeth** | GGA stays advisory (AI = advisory by nature); real blocking comes from deterministic gates (Item 8 linters, Item 2 coverage, Item 4 govulncheck) | Requires the other items to land for teeth | Medium |

### Recommendation
**Approach C (which keeps A for GGA specifically)** — do **NOT** make GGA a hard gate.
The decision to keep GGA non-blocking was deliberate (REQ-CI-004, 2026-06-18) and is
vindicated by repeated GGA infrastructure failures in the archive. GGA is an AI reviewer
with false positives — it belongs in advisory CI. The real enforcement teeth should come
from **deterministic** gates: the new refactoring linters (Item 8), per-package coverage
(Item 2), and blocking govulncheck (Item 4). Conflating AI-review with deterministic
gates is the trap. If a hard gate is truly wanted later, it requires a MODIFIED
REQ-CI-004 delta with explicit rationale — flag this as a spec contradiction otherwise.

### Risks
- Treating GGA as advisory may let AGENTS.md violations merge — Medium. Mitigate: the
  local pre-commit (Item 1) runs GGA before every commit; `NO-VERIFY:` makes bypasses
  honest and traceable; deterministic gates cover what GGA would catch mechanically.
- Contradicting REQ-CI-004 silently — High if done without a delta. Mitigate: any change
  to GGA blocking MUST update the ci-consistency spec.

## Cross-Cutting Recommendation (the through-line)

The orchestrator's 8 gaps conflate two kinds of gates:
1. **Deterministic gates** (lint, coverage, govulncheck, build) — these SHOULD block CI
   and the local pre-commit. Items 2, 3, 4, 8, and the lint/vet/build part of Item 1.
2. **AI review** (GGA) — advisory by nature, enforcement via local pre-commit +
   `NO-VERIFY:` honesty. Item 6.

**This change should add teeth to deterministic gates and leave GGA advisory.** Doing
the opposite (hard-gating GGA) would contradict REQ-CI-004 and block PRs on GGA's own
demonstrated flakiness.

### Suggested scope for the proposal (fits 800-line budget)
- Item 1: versioned `.githooks/pre-commit` + `task setup` + CONTRIBUTING note (~40 lines)
- Item 2: `task cover:pkg` per-internal-80% gate + CI wiring (~30 lines)
- Item 3: `golangci-lint-action@v8` pinned v2.12.2, remove `go install @latest` (~10 lines net)
- Item 4: govulncheck blocks, gosec advisory, pin versions (~15 lines)
- Item 5: `go install gga@v2.8.1` replacing brew (~5 lines)
- Item 6: **no change** (keep advisory; document why in proposal) — or MODIFIED REQ-CI-004 if hard gate wanted
- Item 8: enable maintidx + dupl; fix gitea/github + cmd dupl pairs (~120 lines incl. consolidation)
- Specs: MODIFIED `ci-consistency` (REQ-CI-005 per-package coverage, REQ-CI-006 pinned
  linter action, REQ-CI-007 govulncheck-blocks); possibly new `refactoring-linters` capability

### Risks (consolidated)
- **GGA `go install` path unverified** (Item 5) — Medium. Apply must confirm import path.
- **gitea/github consolidation may conflict with cloud-consolidation archive** (Item 8) — Medium.
- **tui/screens coverage at exactly 80.0%** (Item 2) — Medium. New gate may flake; needs a tiny coverage bump.
- **Spec contradiction if GGA hard-gated** (Item 6) — High. Must update REQ-CI-004.
- **gocognit deferral leaves complexity debt unmeasured** (Item 8) — Medium. Tracked follow-up.

### Ready for Proposal
**Yes.** The orchestrator should tell the user:
- 7 of 8 gaps are confirmed and scoped; the recommended change is ~220 lines + a
  ~120-line cloud/cmd DRY consolidation, well within the 800-line budget.
- **One decision needed from the user:** confirm GGA stays advisory (recommended) vs.
  hard gate (contradicts REQ-CI-004, needs a MODIFIED spec delta). Default: advisory.
- **One apply-phase prerequisite:** verify the GGA `go install` import path before
  changing gga.yml.
- **Refactoring linters are split:** maintidx+dupl now (fix the 3 real DRY pairs),
  gocognit/funlen/nestif deferred to a tracked follow-up (44 violations incl. 3
  architectural SEVERE that need real extraction, not config).
